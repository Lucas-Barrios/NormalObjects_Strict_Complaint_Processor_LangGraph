# Lab Summary — LangGraph vs LangChain

For a stateful, multi-step complaint workflow like Bloyce's Protocol, **LangGraph is the right tool**: it models the process as an explicit directed graph where each node (intake, validate, investigate, resolve, close) is a discrete unit of work with its own state transition, retries, and routing logic. The fan-out/fan-in pattern for parallel investigation, the `interrupt()`-based human-in-the-loop checkpoint, and SQLite-backed persistence across process restarts are all first-class primitives in LangGraph — implementing them from scratch on top of plain **LangChain** would require hand-rolling a state machine, managing concurrency manually, and building your own checkpoint store. LangChain shines for **simpler, linear pipelines**: a single RAG chain, a one-shot tool-calling agent, or a prompt template sequence where there is no branching, no long-running state, and no need to pause mid-run for human input. The core trade-off is **structure vs flexibility** — LangGraph forces you to define nodes, edges, and a typed state schema up front (more boilerplate, steeper learning curve), but that explicitness makes complex workflows testable, observable, and resumable; LangChain's LCEL lets you compose chains quickly with less ceremony, but becomes hard to reason about once conditional branching, parallel execution, or durable state enter the picture.
